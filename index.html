Solution Analysis — PORTAL-58731

Audit MS Kafka ESB Integration

Zheng Zhang · March 2026

The Problem

  • Audit Trail events currently flow: Services → Kafka → Audit MS → Database
  • New requirement: these events also need to reach an external vendor via ESB
  • Question: how do we get audit events from Kafka to ESB?
Current: Services ──→ Kafka ──→ Audit MS ──→ DB ✅ Needed: Services ──→ Kafka ──→ ??? ──→ ESB ──→ Vendor 🆕

Four Solutions Investigated

Solution 1 — ESB Consumes Kafka Directly

Services ──→ Kafka ──→ ESB (new Kafka consumer) ──→ Vendor

✅ Pros

  • Clean separation — Audit MS untouched
  • ESB can scale independently
  • No new infrastructure

⚠️ Cons

  • ESB needs Kafka client + SSL config
  • ESB team: significant work
  • Couples ESB to our Kafka cluster

Solution 2 — Dedicated Kafka Topic

Variant A: Services ──→ Existing Topic + New Topic ──→ ESB Variant B: Existing Topic ──→ Kafka Streams ──→ New Topic ──→ ESB

✅ Pros

  • Best decoupling — fully independent
  • Each system owns its own topic
  • Scales independently

⚠️ Cons

  • New topic + infra setup
  • Variant A: producer changes (dual-write)
  • Variant B: deploy Kafka Streams app
  • ESB still needs Kafka client

Solution 4 — Audit MS + REST → ESB

┌──→ DB (existing consumer, unchanged) Services ──→ Kafka ──┤ └──→ New consumer ──→ REST POST ──→ ESB ──→ Vendor

✅ Pros

  • Minimal changes — no new infra
  • ESB only needs a REST endpoint
  • Reuses existing Kafka config & SSL
  • Fastest to implement

⚠️ Cons

  • Audit MS takes on extra responsibility
  • Cannot scale ESB consumer independently
  • Audit restart affects ESB forwarding

Key Questions for the Team

  • Expected message volume? — Affects whether independent scaling matters
  • ESB team preference? — Kafka client vs REST endpoint is their call
  • Filtering logic? — Which audit events go to vendor? (Needs business input)
  • Failure handling? — Retry policy, dead-letter strategy
  • Long-term architecture? — Start simple and evolve, or build for scale now?
